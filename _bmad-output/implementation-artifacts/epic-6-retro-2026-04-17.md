## 2. What Went Well

### 2.1 Epic 5 foreshadowing paid off exactly as predicted

Story 5.1's forward-loaded expansion of the Todo model (adding `priority`, `deadline`, `category_id` in a single migration) meant Epic 6 started with **zero backend work**. `PATCH /api/todos/{id}` and `POST /api/todos` already accepted both fields. `useUpdateTodo` and `useCreateTodo` already handled optimistic writes for both. The 86-test backend suite remained unchanged. This is the clearest validation yet of the "plan foundational stories to serve N+1 epics" insight from the Epic 5 retro.

### 2.2 Frontend test habit began to take hold

Epic 5 retro action A1 called for unit tests on date/priority logic. Story 6.2 delivered `frontend/src/lib/utils.test.ts` with 10 boundary-case tests covering `formatDeadline` (null, today, tomorrow, 2–6 days, beyond 6, past/overdue), `isOverdue`, and `toISODate`. Frontend test count tripled (6 → 18). The `pnpm test` command passes green. Tooling that landed in Epic 4 finally has a testing habit forming around it.

### 2.3 Pattern reuse compounded across popovers

`DeadlineDatePickerPopover` was built from `PriorityPickerPopover` (Story 6.1), which was itself built from `CategoryPickerPopover` (Story 5.3). Three generations of the same popover pattern — click-outside via mousedown, Escape handling, `requestAnimationFrame` focus, `closedByMouseDownRef` race guard — executed without divergence. `DeadlineLabel` was built from `PriorityIndicator` (Story 6.1). Every component had a living reference to clone from.

### 2.4 Story 6.1 review patches were applied as standing wisdom in 6.2

Story 6.1's code review surfaced three patches (`339ae5f`): always-render transparent 3px border for consistent layout, extract `getPriorityColor` to local const, and `closedByMouseDownRef` race guard. All three were folded explicitly into Story 6.2's Dev Notes (item 11 of "Previous Story Intelligence") and applied proactively in `DeadlineLabel` without needing to rediscover them. This is Previous Story Intelligence (C1) working exactly as intended — patches become institutional memory.

### 2.5 Zero new dependencies, native primitives throughout

Date picker: native `<input type="date">`. Priority dropdown in FAB: native `<select>`. Popover positioning: absolute + z-50 + manual focus. No `react-datepicker`, no `framer-motion`, no `radix-ui/popover`. Bundle stayed lean. The "no JS animation libraries" constraint from the architecture paid off in keeping third-party surface area zero.

### 2.6 Scope boundaries held across the epic

Story 6.1's "What NOT to Do" explicitly deferred DeadlineLabel and overdue treatment to 6.2; 6.2 explicitly deferred ViewSwitcher and DeadlineGroupHeader to Epic 7. Both stories refused to modify `use-todos.ts`, `types/index.ts`, or backend code because all three were already prepared by Epic 5. Zero scope creep.

### 2.7 Low review noise, high signal

Only 2 actionable patches across 2 stories (3 line-level fixes in 6.2 patches). No architectural rework. Review gate remains clean.

## 3. Challenges

### 3.1 Popover composition gap surfaced only post-merge

The biggest finding of Epic 6 was **not in either story** — it was in the UI polish commit `bfa0d62` after merge:

- **Popovers clipped by parent `overflow-hidden`.** The priority/deadline/category inline-edit popovers were clipped by their containing `CategorySection` when that section used `overflow-hidden` in its collapse animation. The fix was to change popovers to open upward (`bottom-full`) and make `CategorySectionHeader` use `overflow-visible` when expanded.
- **FAB panel felt cramped** with three selectors side-by-side; selectors had to be restacked vertically with aligned labels.
- **Gear icon for Category management was not discoverable.** Replaced with a labeled "Category" FAB button next to the "Todo" FAB.

None of the 6.1 or 6.2 story specs required the dev to test **composition** — priority indicator + deadline label + category chip together, popover opening inside a scrollable/collapsing container, three FAB selectors at once. The individual components all worked in isolation. The assembly did not.

**Root cause:** Story Dev Notes specified each feature in isolation; no explicit acceptance criterion said "verify popovers are not clipped when opened inside CategorySection when collapsed/animating" or "verify three FAB selectors fit on mobile."

### 3.3 Component-level tests still absent

`utils.test.ts` is pure-logic. Epic 6 added 4 new interactive components (`PriorityIndicator`, `PriorityPickerPopover`, `DeadlineLabel`, `DeadlineDatePickerPopover`) — all with click-outside logic, keyboard handling, and the `closedByMouseDownRef` race guard. **Zero component tests.** The race guard bug was caught twice by manual code review (6.1 and 6.2) — a component test for "click indicator, click outside, click indicator again" would catch it automatically.

### 3.4 PriorityIndicator color-mapping never got the unit test it asked for

Epic 5 retro A1 specifically called out *"PriorityIndicator color mapping"* as a test candidate. `utils.test.ts` covers deadline helpers but not `getPriorityColor()` or `PRIORITY_LEVELS`. The test file was opened by Story 6.2; Story 6.1 could have added priority helper tests first.

### 3.5 The "Today" color fix in 6.2 patches was a spec ambiguity

Story 6.2's spec said "Today" uses `--color-text` (standard) + bold weight. The initial implementation used `--color-text-muted` for all non-overdue states. Patch `a31a901` switched "Today" to foreground. This was a spec-to-implementation slip (the table in Dev Notes was right; the code matched the wrong precedent). Small, but a reminder that tables in Dev Notes are source of truth only if the dev reads them against the actual code path.

## 4. Key Insights

1. **Composition is the missing acceptance criterion.** Epic 6 stories specified each feature in isolation with airtight per-feature ACs. The bugs that remained — popover clipping, FAB spacing, selector overflow on mobile — all appeared only when features composed. Future story Dev Notes should include an explicit "Composition Checks" section listing the interactions to manually verify: popover × collapsing-parent, FAB × mobile × all-selectors-populated, etc.

2. **Tests that would have caught the real bugs are component tests, not logic tests.** The race guard (`closedByMouseDownRef`) bug was caught twice by manual review across 6.1 and 6.2. A unit test for pure helpers is valuable but doesn't exercise the interaction paths where bugs actually live. Epic 7 is the moment to add the first RTL component test.

3. **Standing action items decay silently.** A3 (temp-ID DELETE race) has been on the list since Epic 3. Without a mechanism to surface "items older than one epic," they're invisible at planning time. A "Carried Debt" section at the top of each new story's Dev Notes would keep them visible.

4. **Forward-loading works across two epic boundaries.** Story 5.1's metadata expansion served Epic 5 **and** Epic 6. The cost was small (3 nullable columns, one migration); the savings were large (zero backend work in Epic 6). The same question is now worth asking for Epic 7: *is there any client-side selector/filter utility that 7.1 could build as a reusable primitive for 7.2 instead of duplicating grouping logic?* The `formatDeadline` helper from 6.2 is already positioned to power DeadlineGroupHeader in 7.2 — a mini-example of the same pattern.

5. **Post-release UI polish commits are a retro signal.** `bfa0d62` is not a bug — it's useful work. But it's also evidence that the epic's acceptance gate doesn't verify assembly, only features. Treating these commits as retro input (not ignored as "polish") is how the gap closes.
