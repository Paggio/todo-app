## 2. What Went Well

### 2.1 Forward-thinking foundational story (5.1) eliminated Epic 6 backend work

Story 5.1 didn't just build the category backend — it expanded the Todo model with `deadline` (date, nullable), `priority` (integer 1-5, nullable), and `category_id` (FK, nullable) fields. This means Epic 6 (Priority & Deadline Management) has **zero backend prep work**. The API already accepts `priority` and `deadline` in create and update endpoints, and `useUpdateTodo` already handles them in optimistic cache writes. This is the strongest example yet of a foundational story paying dividends beyond its own epic.

### 2.2 Zero new npm dependencies across the entire epic

The category management panel, slide-in animations, frosted glass backdrop, CategorySectionHeader, CategoryChip, and CategoryPickerPopover were all built with existing CSS tooling (keyframes, transitions, Tailwind utilities) and native browser APIs. No new libraries installed. This keeps the bundle lean and avoids dependency churn.

### 2.3 Three-step optimistic pattern replicated flawlessly

`use-categories.ts` (Story 5.2) replicated the three-step optimistic mutation pattern from `use-todos.ts` (Epic 3) verbatim — `onMutate` (cancel queries → snapshot → optimistic write → return context) → `onError` (guarded rollback + toast) → `onSettled` (invalidate). Four hooks (useGetCategories, useCreateCategory, useRenameCategory, useDeleteCategory) all follow the pattern. The critical detail — `useDeleteCategory.onSettled` invalidating **both** `["categories"]` and `["todos"]` — was correctly implemented.

### 2.4 Cross-story pattern reuse: CompletedSection → CategorySectionHeader

Story 5.3's `CategorySectionHeader` was modeled directly on the existing `CompletedSection` collapse pattern (localStorage key, `aria-expanded`, chevron toggle, 200ms animation). The Dev Notes explicitly cited the pattern. This reduced implementation time and ensured UX consistency.

### 2.5 Review gate remained clean

Only 1 actionable patch across 3 stories — the stale `selectedCategoryId` after category deletion in FAB (Story 5.3). Three other findings dismissed as noise. This is a high signal-to-noise ratio for an epic that created 6 new files and modified 6 existing ones.

### 2.6 Scope boundaries held perfectly

Every story's "What NOT to Do" section enforced strict separation:
- Story 5.1: "Do NOT add `use-categories.ts` — that is Story 5.2"
- Story 5.2: "Do NOT add category assignment to todos — that is Story 5.3"
- Story 5.3: "Do NOT add ViewSwitcher — that is Epic 7"

Zero scope creep across the epic. The C2 convention from Epic 3's retro is fully institutionalized.

### 2.7 Backend test growth was substantial and well-structured

47 → 86 tests. Story 5.1 added 22 category-specific tests (CRUD, isolation, duplicate name, cascade delete) and 17 expanded todo field tests (category_id, deadline, priority in create/update). Test patterns (`_register_and_login`, `_assert_error_envelope`) were reused without modification.

## 3. Challenges

### 3.1 Frontend test coverage remains minimal despite vitest installation

Vitest was installed in Epic 4 (action item A1 from Epic 3 retro). However, the only test file is `login.test.tsx` with 6 tests. Epic 5 added:
- 4 new mutation hooks with optimistic logic (`use-categories.ts`)
- 3 new interactive components (`CategorySectionHeader`, `CategoryChip`, `CategoryPickerPopover`)
- Modified FAB with category dropdown and session memory
- Refactored `home.tsx` with category grouping logic

**Zero frontend tests for any of it.** The tooling landed; the habit didn't. This is compounding debt — Epic 6 will add date-relative logic (DeadlineLabel), priority color rendering, and inline editing popovers, all building on an untested foundation.

### 3.2 ESLint friction across Stories 5.2 and 5.3

Three distinct ESLint issues required debug time:
1. `react-hooks/set-state-in-effect` (5.2) — forced refactoring panel state sync from `useEffect` to in-render derivation
2. `react-hooks/refs` (5.2) — caught ref-based sync pattern that needed removal
3. `useCallback` dependency on recreated `options` array (5.3) — fixed with `useMemo`

Each was a legitimate catch (not false positives), but the pattern recurrence suggests the dev agent would benefit from explicit ESLint pattern documentation in story Dev Notes.

### 3.3 Pre-existing lint/build errors carried forward again

`login.tsx` has 2 refs-during-render warnings and `login.test.tsx` has type mismatches. Noted as "out of scope" in Stories 5.2 and 5.3. These have been carried since Epic 4. Low severity individually, but accumulating age signals broken-window debt.

### 3.4 Epic 4 retrospective was skipped

Epic 4's retrospective was marked "optional" and never run. This broke the accountability chain on Epic 3's 7 action items:
- 4/7 fully completed (C1, C2, A3, A5)
- 1/7 partially completed (A1 — tooling installed, habit not established)
- 1/7 unverifiable (A2 — a11y consolidation in Story 4.8)
- 1/7 missed entirely (A4 — temp-ID DELETE race fix)

Three consecutive retros with 100% follow-through (Epic 1→2, 2→3) — then a skip, and follow-through dropped to ~60%. **Retrospectives are load-bearing for accountability.**

### 3.5 SQLModel ORM quirk discovered at implementation time

Story 5.1 hit a TypeError with `sa_column_kwargs={"ondelete": "SET NULL"}` — SQLModel doesn't pass `ondelete` as a column kwarg. Resolved by removing the kwarg and relying on the Alembic migration + explicit router-level cascade for SQLite test compatibility. Minor impact but unpredictable from planning.

## 4. Key Insights

1. **Forward-loading foundational stories across epic boundaries is high-leverage.** Story 5.1 expanded the todo model for Epic 6's needs, not just Epic 5's. The incremental cost was small (3 nullable columns + schema changes), but Epic 6 now starts with zero backend work. **Plan foundational stories to serve N+1 epics, not just the current one.**

2. **Skipping retrospectives has measurable cost.** Epic 4's skip resulted in 1 missed action item, 1 partial completion, and 1 unverifiable item. The retro isn't ceremony — it's the mechanism that converts lessons into commitments and commitments into follow-through.

3. **ESLint friction is a leading indicator, not a problem.** Three catches in two stories prevented real bugs (render loops, stale refs, unstable dependencies). The friction is worth documenting as institutional knowledge so future stories avoid re-discovery.

4. **Frontend test tooling ≠ frontend test culture.** Installing vitest (A1) was necessary but insufficient. The habit of writing tests alongside features never took hold. Epic 6 is the right moment to establish the habit — `DeadlineLabel`'s date-relative logic is a perfect unit test candidate.
