## Deferred from: code review of 1-2-initialize-backend-with-fastapi-and-docker (2026-04-15)

- **Container runs as root** (`backend/Dockerfile`) — No `USER` directive; the API process runs as root inside the container. Real security concern for production hardening, out of scope for dev scaffold story.

## Deferred from: code review of 2-1-user-registration (2026-04-15)

- **passlib 1.7.4 is end-of-life; bcrypt pinned to 4.x as workaround** (`backend/requirements.txt`) — bcrypt 5.x removed `__about__` and added strict length validation that breaks passlib 1.7.4. Workaround is stable but the library is unmaintained. Long-term: replace with direct `bcrypt` usage or migrate to passlib 2.x when GA.
- **`users.created_at` column is timezone-naive** (`backend/app/models/user.py`, `backend/alembic/versions/7e729bfa2ac7_create_users_table.py`) — SQLAlchemy `DateTime()` without `timezone=True`. Current inserts use `datetime.now(timezone.utc)` so values are correct, but the column type doesn't enforce UTC. Swap to `DateTime(timezone=True)` in a schema-hardening pass or the first migration that naturally touches the table.
- **Alembic naming convention not configured on `SQLModel.metadata`** (`backend/alembic/`) — every autogenerated index will default to `ix_<table>_<col>` instead of the architecture-spec `idx_<table>_<col>`. Story 2.1's migration required a manual rename. Configure `MetaData(naming_convention={...})` on SQLModel's underlying metadata so future migrations emit the correct names automatically.
- **No CSRF token protection on cookie-authenticated endpoints** (`backend/app/routers/auth.py`) — cookies are set with `samesite=lax` which blocks cross-site POSTs and is sufficient for a single-domain app. For deployment on a shared domain or with subdomain trust concerns, add a CSRF token pattern (double-submit cookie or synchronizer token).
- **Race condition on concurrent duplicate-email registration** (`backend/app/routers/auth.py`) — two simultaneous `POST /register` with the same email can both pass the `SELECT` existence check; one will fail at `commit()` with an `IntegrityError` that currently bubbles as a 500 instead of the expected 409. Low probability for a single-user app; add an `IntegrityError` handler that remaps to 409 + `EMAIL_ALREADY_EXISTS` if this becomes a real concern.

## Deferred from: code review of 1-4-project-readme-and-documentation (2026-04-15)

- **`shell-backend` /bin/sh assumption** (`Makefile:38`) — `python:3.12-slim` includes `/bin/sh`; safe now but undocumented assumption. If base image changes, target breaks silently.
- **Frontend healthcheck tight coupling to backend `/` endpoint** (`docker-compose.yml`) — Frontend depends on backend healthcheck which pings `http://localhost:8000/`. If that endpoint changes or is removed, frontend startup will hang indefinitely.
- **Anonymous volumes (`node_modules`, `.venv`) purged on `docker compose down`** (`docker-compose.yml`) — These anonymous volumes are recreated on every `up`; they are not the same as the named `postgres_data` volume. Documentation only promises persistence for `postgres_data`.
- **Backend `entrypoint.sh` lacks migration retry logic** (`backend/entrypoint.sh`) — `alembic upgrade head` runs immediately after db healthcheck passes with no retry for transient connection failures on a loaded host.
- **`CORS_ORIGIN` single-origin behavior undocumented** (`backend`) — Backend enforces a single CORS origin. Behavior for multiple-origin configuration (comma-separated, etc.) is undefined and undocumented.

## Deferred from: code review of 7-1-view-switcher-and-due-this-week-view (2026-04-17)

- **`useDeleteTodo` optimistic-create/delete race not fully closed by A4 guard** (`frontend/src/hooks/use-todos.ts:155`) — The new `id < 0` early-return correctly stops the 404 round-trip, but `onSettled` still invalidates `["todos"]`. If the pending POST from `useCreateTodo` resolves after the user's optimistic delete, the subsequent GET refetch pulls the server-created todo back into the cache and the "deleted" item reappears. A full fix would need to either abort the pending create or chain a real delete once the create resolves. Low-frequency edge case (requires sub-second user interaction) and the current guard is still a net improvement over the prior behavior.
- **`selectDueThisWeek` "today" boundary captured inside memo** (`frontend/src/hooks/use-todos.ts:471`, `frontend/src/pages/home.tsx:67`) — `new Date()` is evaluated inside the function, and `HomePage` wraps the call in a `useMemo` keyed only on `todos`. If the user leaves the app open past local midnight without any todo mutation, the week boundary does not advance. Any mutation re-runs the memo and self-heals; a full fix would be a time-of-day ticker or stale-interval invalidation. Low severity.
