# Story 2.1: User Registration

Status: done

## Story

As a visitor,
I want to register a new account with my email and password,
so that I can start using the todo application.

## Acceptance Criteria

1. **Given** the `User` SQLModel (`id`, `email`, `hashed_password`, `created_at`) and its Alembic migration are created **When** a visitor sends `POST /api/auth/register` with a valid email and password **Then** the system creates a new user with the password hashed using bcrypt (cost factor >= 10), returns the user object (without `hashed_password`), and sets a signed JWT (HS256, 7-day expiry) in an httpOnly cookie (NFR5, NFR6)

2. **Given** a visitor sends `POST /api/auth/register` with an email that already exists **When** the server processes the request **Then** it returns **409** with `{ "detail": "Email already registered", "code": "EMAIL_ALREADY_EXISTS" }`

3. **Given** a visitor sends `POST /api/auth/register` with an invalid email or missing password **When** the server processes the request **Then** it returns **422** with `{ "detail": "...", "code": "VALIDATION_ERROR" }`

4. **Given** the frontend auth screen is displayed **When** a visitor fills in the registration form (email + password) and submits **Then** on success the user is redirected to the main todo view (`/`); on error an inline error message is displayed below the form

5. **Given** the auth screen component **When** it renders **Then** it supports toggling between **Sign In** and **Sign Up** modes (Sign In form wiring is Story 2.2; the toggle and layout land here)

## Dev Notes

### Critical Architecture Constraints (must follow)

- **httpOnly cookie — the frontend NEVER reads the token.** No `document.cookie` parsing, no `localStorage`, no token in React state. The cookie is set by the backend `POST /api/auth/register` response and included automatically by `fetch({ credentials: "include" })`. [Source: architecture.md#Authentication & Security L184-189; #Anti-Patterns L361]
- **API JSON is snake_case; TypeScript is camelCase.** The transformation boundary is `lib/api.ts`. Every component and hook consumes camelCase; every backend test and curl example sees snake_case. [Source: architecture.md#API Boundary Rule L276]
- **Error response contract:** `{ "detail": "...", "code": "..." }` — flat shape, not FastAPI's nested default. The story's Task 5.3 installs the handler that enforces this. [Source: architecture.md#Error Codes L303-304]
- **bcrypt cost factor >= 10.** Passlib defaults are safe; the story sets cost 12. Do not reduce below 10. [Source: architecture.md#Password hashing L187; NFR5]
- **JWT is HS256, 7-day expiry, signed with `settings.jwt_secret`.** No refresh token. [Source: architecture.md#Token type/expiry L184-186]
- **Per-user isolation is NOT this story's job.** `get_current_user` and `user_id` scoping land in Story 2.3. Do not build them speculatively — leave the `TODO(Story 2.3)` marker in `core/deps.py`.
- **Three-step optimistic mutation pattern is NOT required here.** Registration is a one-shot write with no client cache to roll back. `useMutation` without `onMutate`/`onError`/`onSettled` is correct. The optimistic pattern starts mattering in Epic 3. [Source: architecture.md#Optimistic Mutation Pattern L318-324]
