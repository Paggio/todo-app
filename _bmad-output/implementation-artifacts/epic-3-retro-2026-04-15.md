- **T2 (`DateTime(timezone=True)`):** `todos.created_at` uses `sa.DateTime(timezone=True)` from day one. **Bonus:** the same migration included `op.alter_column('users', 'created_at', type_=sa.DateTime(timezone=True))` — Story 3.1 opportunistically paid down the Epic 2 `users.created_at` debt. **Two debt items retired in one story.**
- **T3 (`test_todos.py` parity):** 18 new tests in Story 3.1 mirror `test_auth.py` depth — isolation (3), 404 (2), auth guard (4), validation (2), CRUD happy-paths (6), response shape (1). `_assert_error_envelope` helper reused verbatim.
- **T4 (Three-step optimistic pattern Dev Notes at Story 3.3):** Documented with exact `onMutate` (cancelQueries → snapshot → optimistic write → return context) → `onError` (guarded rollback + toast) → `onSettled` (invalidate) spec. Stories 3.4 and 3.5 both cite it directly and follow it without deviation. **The Dev Notes became a durable reference template.**

**Three retros in a row with 100% action-item completion:** Epic 1→2: 4/4, Epic 2→3: 6/6. This is a mature, proven mechanism.

### 2.2 Cross-story review learning within a single epic

Story 3.3's code review flagged 5 patch items — one was the `onError` undefined-context bug (if `onMutate` throws before producing a snapshot, `context?.previousTodos` is undefined and `setQueryData(["todos"], undefined)` clears the cache). Story 3.4's Dev Notes explicitly cited this finding: *"Learned from Story 3.3 code review finding #3."* Story 3.4 then guarded the same pattern, as did Story 3.5. **Lessons propagated within a single epic, not just across epics** — tighter-grained learning than Epic 2 achieved.

### 2.3 Net deferred-work delta from Story 3.1 was negative

Story 3.1 retired two debt items (alembic naming convention, `users.created_at` tz-naive) while introducing only four new ones (no FK CASCADE on `todos.user_id`, placeholder revision ID, no `server_default` on `is_completed`, fragile test ordering) — all minor hardening items tied to pre-deploy or user-deletion features that do not exist yet.

### 2.4 The `["todos"]` query key contract held across five stories

Chosen in Story 3.2 for `useGetTodos`. Three mutations (3.3, 3.4, 3.5) invalidate it. `useLogout` from Story 2.3 already referenced it forward-compatibly. Zero collisions, zero rename churn across the epic.

### 2.5 Review gate matured

Epic 2 already used three-layer review (Blind Hunter / Edge Case Hunter / Acceptance Auditor). Epic 3 kept that standard and Story 3.3 went through two review passes — the second pass verified all 5 patch fixes landed, then produced a clean gate with only one pre-existing defer. Mechanism is now a repeatable template.

### 2.6 Calm error UX landed correctly

Story 3.6 resisted the obvious temptation to use `text-destructive` (red) on the error message. The UX spec says "errors are conversations, not alarms" — the implementation uses `text-muted-foreground` with a "Try again" underlined link. **This is the first story where UX language was respected over engineering defaults** — a signal that design polish work in Epic 4 will have better baseline collaboration.

## 3. Challenges

### 3.1 Zero automated frontend tests across the entire epic

Story 3.2 noted "no vitest/jest configured; tests deferred to when framework is set up." Every subsequent frontend story repeated the same note. At end of Epic 3, the frontend surface includes:

- 4 mutation hooks with nontrivial optimistic logic (cancel queries, snapshot, optimistic write, guarded rollback, invalidate)
- 6+ interactive components (FAB, TodoItem, TodoList, CompletedSection, EmptyState)
- Keyboard, ARIA, and focus-management state across the interactive surface

…and zero automated unit tests. Manual Playwright smokes + `pnpm typecheck` + `pnpm lint` are the only safety net. **This is the Epic 3 debt item most likely to bite in Epic 4**, which has 8 stories of visual/interactive polish touching these same components. Addressed by action item A1.

### 3.2 Accessibility debt accumulated and scattered across multiple stories

Five deferred entries spanning four stories:
- `aria-controls` on conditionally-rendered element (from 3.2, repeated in 3.4 review)
- Empty `<div role="list">` announcement (resolved by 3.6's EmptyState, but only after it was left unresolved for two stories)
- No `aria-live` for deleted items (3.5)
- "Try again" button lacks loading indicator during refetch (3.6)
- "tap +" copy in EmptyState implies touch-only (3.6)

Scattered across review findings rather than consolidated. Story 4.8 is the natural collection point, but without deliberate aggregation we risk rediscovering each item one-by-one in review. Addressed by action item A2.

### 3.3 Optimistic-create → fast-delete race has a latent edge case

If the user deletes a freshly-created todo before the server responds (the todo still has a negative temp ID from `useCreateTodo`), `DELETE /api/todos/-1713...` fires and returns 404. Rollback + revalidation handle it gracefully, but there's a brief flash. Edge case requiring sub-second user interaction. Addressed by action item A4 at Story 4.4.

### 3.4 `useUpdateTodo` optimistic write only handles `isCompleted`

If description editing is added later (Epic 4's component polish stories could introduce inline edit), `onMutate` needs to extend. Flagged in Story 3.4 review. Not in Epic 3's scope, but a documented forward-looking concern.

### 3.5 Manual smoke tests frequently deferred to human

Across Stories 3.3, 3.4, 3.5, 3.6, subtasks for browser-level Playwright verification were marked "deferred to human reviewer" because the dev agent couldn't invoke Docker. Creates an implicit trust boundary: did the user actually run every deferred smoke? Without automated frontend tests (3.1 above), this gap compounds.

### 3.6 `errors.py` package-root precedent stands unreviewed

Flagged in Epic 2 retro. Epic 3 didn't add new cross-cutting modules, so it didn't come up again. The precedent is now one epic older without being either ratified or corrected.

## 4. Key Insights

1. **Templates from Story 3.3's Dev Notes were higher-leverage than architecture docs.** The three-step optimistic pattern written in-story + its code was more authoritative for Stories 3.4/3.5 than architecture.md's prose description. **In-story references beat external docs for cross-story propagation** — second epic confirmation.

2. **One foundational story + N template applications is a reliable epic shape.** Epic 1's 1.1 was the shakedown. Epic 2's 2.1 was the auth foundation. Epic 3 had two: 3.1 carried the backend CRUD + tests, 3.3 carried the optimistic frontend pattern. Both paid their weight and unlocked multiple follow-ups. **Expect Epic 4 to show the same shape** — 4.1 (design tokens) is the foundational story.

3. **Cross-story learning is faster when stories share a module.** Stories 3.3 → 3.4 → 3.5 all edit `use-todos.ts`. Review-finding propagation was crisp (the `onError` guard). This is tighter than Epic 1/2's cross-file learning.

4. **The "defer to Epic 4" pattern only works if Epic 4's collection-point story is authored with that in mind.** Every frontend story properly punted a11y polish to Story 4.8 — but that only pays off if 4.8 is planned with explicit awareness of the 5+ deferred items to consolidate, not as a greenfield "do accessibility."
