## 2. What Went Well

### 2.1 The "Previous Story Intelligence" pattern

Every story file after 1.1 started with an explicit "Previous Story Intelligence" section that synthesized lessons from prior stories. This is the epic's strongest process invention.

- Story 1.2 captured Docker daemon instability and the kebab-case vs snake_case convention difference from 1.1.
- Story 1.3 caught the frontend lockfile drift caused by 1.1's review patch *before* the build actually failed.
- Story 1.4 synthesized all three priors and was the smoothest story of the epic.

**This pattern is confirmed working and is retained as a standing story-file convention.**

### 2.2 Defer-first review discipline

The Senior Developer review agent consistently caught "happy path only" implementations (empty env-var bypass, unhandled `KeyError` in `alembic/env.py`, missing restart policies, `depends_on` without health condition). Each finding has a clear `[Review][Patch]` or `[Review][Defer]` disposition and deferred items are traceable to source stories via `deferred-work.md`. Debt is visible, not hidden.

### 2.3 Velocity acceleration after Story 1.1

Story 1.1 (2026-04-14) was the shakedown. Stories 1.2, 1.3, and 1.4 all completed on 2026-04-15 — Story 1.3 being the most complex (cross-service orchestration, Alembic setup, hot-reload validation) yet delivered same-day. The "Previous Story Intelligence" pattern is the most likely cause.

### 2.4 Structural patterns worth keeping

- Module-level `settings = Settings()` in `app/core/config.py` for fail-fast Pydantic validation at import time.
- Consistent API error format: `{"detail": "...", "code": "..."}` established as the standard.
- Naming convention boundaries (kebab-case frontend, snake_case Python, camelCase TS, PascalCase React) documented and enforced.

## 3. Challenges

### 3.1 Docker Desktop instability (2 occurrences)

Docker daemon went down during container verification in Stories 1.1 and 1.2, forcing workarounds (validating via local Vite dev server; validating inside partial Docker builds). It earned a permanent troubleshooting entry in the README but is a lived friction point.

### 3.2 Lockfile drift after review patches — only cross-story regression of the epic

Story 1.1's review moved `shadcn` from `dependencies` to `devDependencies`, but `pnpm-lock.yaml` was not regenerated in the same patch. Story 1.3's first `docker compose up --build` failed with `ERR_PNPM_OUTDATED_LOCKFILE`. **This is a process gap, not a one-off miss.** Addressed in action items.

### 3.3 Zero automated tests entering Epic 2 (auth)

`backend/tests/` directory exists but is empty. All Epic 1 verification was manual (`curl`, Docker commands, DOM inspection). Acceptable for infrastructure scaffolding stories; risky for Epic 2, which introduces password hashing, JWT signing, and 401-redirect flows.

### 3.4 Security debt cluster (deferred by design)

4 of the 14 deferred items are security-related: root container, weak JWT_SECRET default, weak Postgres credentials, `.env` git-ignore verification. **Project lead call: local-only scope, defer until pre-deployment.** Tracked in action items with explicit pre-deploy gate.

---

## 4. Key Insights

1. **Forward-propagating knowledge beats documentation.** "Previous Story Intelligence" in-story sections outperformed any external doc as a mechanism for learning between stories.
2. **Scaffolding output is a starting point, not a finished product.** Both shadcn/ui and FastAPI/Pydantic scaffolds produced code that required first-pass fixes (ESLint overrides, empty-string validators, dependency group corrections).
3. **Defer counts going up is not quality going down.** Defer count rose from 0 → 2 → 7 → 5 across the epic. This tracks growing production-awareness, not declining rigor.
4. **"Local-only" is a scope decision, not a quality decision.** Explicitly calling it out enables clean deferral of security hardening without ambiguity.
